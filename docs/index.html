Section / 01 / Overview

What it is

Instrumentality-MCP creates a knowledge/ folder in your project and keeps it synchronized with your codebase — in both directions. Code changes are detected and surfaced for KB review (code→KB); KB spec changes are flagged for developers to verify the implementation still matches (KB→code).

Instead of scattering product context across Notion, Confluence, and README files, everything lives as structured Markdown beside your code — versioned in git, loaded automatically into your agent's context when relevant.

knowledge/ lives in git
Structured Markdown next to your source. Versioned, diffable, reviewable in PRs — never a separate wiki to drift from.
Bidirectional drift
Code changes flag stale KB; KB changes flag stale code. Queues are PM-gated — nothing auto-rewrites itself.
No API keys
The MCP server returns prompts and context. Your agent does the reasoning. Bring whichever model you already pay for.
Section / 02 / Capabilities

Twenty-two MCP tools, four categories

Every tool the kb-mcp server exposes — grouped the same way the in-editor Capabilities panel groups them. Each tool is callable by an MCP-compatible agent, and the read-only tools also back the sidebar dashboards in the editor extensions.

CAT::01 · SYNC

Sync

Detect code↔KB misalignment and coordinate the sync state surfaced in the dashboard. Bidirectional drift, standards conformance, submodule coordination.

kb_drift kb_conform kb_sub kb_status
CAT::02 · AUTHORING

Authoring

Write, scaffold, import, and tag KB content. Two-phase tools that load context first, let the agent fill the template, then write — so new content fits what's already there.

kb_write kb_scaffold kb_import kb_export kb_autotag kb_autorelate
CAT::03 · GOVERNANCE

Governance

Conform code to standards, bootstrap on legacy projects, migrate after template changes, and upgrade the KB schema. Standards-as-data, queue-driven enforcement.

kb_init kb_analyze kb_extract kb_migrate kb_upgrade kb_inventory
CAT::04 · INTROSPECTION

Introspection

Read-only queries over the KB and its history — fetch context for a working file, ask questions, trace impact, inspect schemas, bridge to issue trackers.

kb_get kb_ask kb_impact kb_history kb_schema kb_issue

Section / 03 / Lifecycle

How drift detection works

Two git hooks write drift entries automatically — pre-push when you push, post-merge after merges (so cross-branch semantic conflicts surface even without a file-level git conflict).

Entries land in sync/code-drift.md or sync/kb-drift.md as a structured queue. A PM or developer reviews each entry through their agent (which fetches the diff and explains it in plain English), then decides: summarise into KB, revert the code, or confirm the code already matches.

Every resolution lands in an append-only sync/drift-log/ — so the history of how the KB drifted, and how it was reconciled, is itself versioned alongside the code.

Section / 04 / Extensions

Two editors, one sync surface

The same MCP server backs companion extensions for VSCode and Obsidian. Both render the queue state live — pipeline strip, drift entries with inline verdicts, conform-pending lists, lint diagnostics — and route resolution actions back through the MCP.

VSCode extension

EDITOR :: vscode 1.85+
  • Pipeline strip across code-drift, kb-drift, conform-pending, lint — a glance at sync health.
  • Sidebar accordion with grouped queue entries; click to see diff, file links, and rule context.
  • Inline verdicts: applied, exempted, promoted, dismissed — submitted straight to the MCP.
  • Capabilities panel documenting every MCP tool with natural-language example prompts.
  • Lint diagnostics surfaced into the VSCode Problems view for in-context fixes.

Obsidian plugin

EDITOR :: obsidian 1.4+
  • The same sync surfaces — rendered inside an Obsidian vault opened on your knowledge/ folder.
  • Dashboard view: drift queues, conform-pending, promotions ledger, submodule status.
  • Click any KB target or rule link to jump to the source file in the vault.
  • Submodule branch status pinned at the top with at-a-glance health dots.
  • Lint surfacing for standards files — invalid YAML and structural issues caught before writing.
Section / 05 / Quickstart

Three steps

Get the MCP running locally, point your agent at it, and bootstrap your first knowledge/ folder. Everything below is verbatim from the README.

1

Install

Clone the repo and install the MCP server's Node dependencies.

$ git clone https://github.com/DEONSKY/project-instrumentality kb-mcp
$ cd kb-mcp/knowledge/_mcp && npm install
2

Configure your agent

Point your MCP-compatible agent at server.js using its absolute path. The same config shape works for Claude Code (.claude/mcp.json) and Cursor (.cursor/mcp.json).

{
  "mcpServers": {
    "kb": {
      "command": "node",
      "args": ["/absolute/path/to/kb-mcp/knowledge/_mcp/server.js"]
    }
  }
}
3

Bootstrap

Ask your agent to initialize the knowledge base — or call kb_init directly.

// Via your agent:
"Initialize the knowledge base for this project"

// Or non-interactively:
kb_init({ interactive: false, config: { projectName: "MyApp" } })
Section / 06 / Why

Why this shape

Most "agent knowledge" tooling assumes the agent is the source of truth — embedding everything into an opaque vector store, asking you to trust similarity scores, and silently rewriting documents on your behalf. Instrumentality inverts that.

The MCP server returns prompts and context, not generated content. Your agent does the reasoning, using whichever model you already pay for. There are no API keys to manage, no third-party services to trust, and no hidden state — every change to the knowledge base is a git commit you can review.

The KB lives next to your code. Specs, flows, schemas, ADRs, standards, validation rules — all of them as structured Markdown under knowledge/. Branchable, diffable, mergeable. When the code changes, drift detection surfaces what needs review. When the KB changes, it surfaces what code may now be stale. The agent helps you reconcile; it never silently decides.
