## Why this file exists

The queue files under `sync/` are auto-generated by `kb_drift` and `kb_conform`. They are mostly empty between runs, so a new contributor opening them has no way to learn the entry format. This file shows one realistic entry per queue type, drawn from the actual writers in [_mcp/tools/drift/queue.js](../_mcp/tools/drift/queue.js) and [_mcp/tools/conform/queue.js](../_mcp/tools/conform/queue.js).

---

## `code-drift.md` — code changed, KB may be stale

Written by `kb_drift` Phase 1 (direction: code→KB). One entry per KB file that maps to the changed code paths via `code_path_patterns` in `_rules.md`.

```markdown
## specs/features/billing.md
<!-- fingerprint: a3f9c2e1 -->

- **KB target:** `specs/features/billing.md`
- **Code files:**
  - `src/billing/invoice-generator.ts` since `e8c4f12`
  - `src/billing/tax-calculator.ts` since `e8c4f12`
```

**Heading** = the KB target file. **Fingerprint** = stable hash of (target + code paths + baseline); used to dedupe across baseline updates. **`since`** = the SHA where the change first appeared in the diff window.

A renamed code file appears as:

```markdown
## specs/features/billing.md

- **KB target:** `specs/features/billing.md`
- **Code files:**
  - `src/billing/invoice.ts` ← renamed from `src/billing/old-invoice.ts` since `b1d77a4`
```

Resolution verbs: `summaries` (KB updated), `reverted` (code change was wrong), `dismissed` (with reason).

## `kb-drift.md` — KB changed, code may be stale

Written by `kb_drift` Phase 1 (direction: KB→code). One entry per KB file whose content shifted in a way that may invalidate the implementation.

```markdown
## specs/features/billing.md
<!-- fingerprint: 4b21d8e9 -->

- **KB file:** `specs/features/billing.md`
- **Code areas to review:**
  - `src/billing/**`
  - `src/api/billing-routes.ts`
```

A KB rename appears as:

```markdown
## specs/features/subscription-billing.md

- **KB file:** `specs/features/subscription-billing.md` ← renamed from `specs/features/billing.md`
- **Broken wikilinks (3 files reference the old path):**
  - `specs/flows/checkout.md`
  - `decisions/payment-provider-choice.md`
  - `integrations/stripe.md`
- **Code areas to review:**
  - `src/billing/**`
```

If a KB file has no matching `code_path_patterns` entry, the entry includes:

```markdown
- **⚠ Unmapped:** no code_path_patterns entry — drift detection inactive for this KB file
```

This is the same warning that appears in the drift-log (see existing [drift-log/2026-04.md](drift-log/2026-04.md)).

Resolution verb: `kb_confirmed` (developer reviewed code, confirmed match) or `dismissed`.

## `standards-drift.md` — current-diff conformance violations

Written by `kb_conform` Phase 1. The queue key uses dotted notation: `<standard-id>.<rule-id>`.

```markdown
## baseline-resolution.never-default-to-head

- **Standard:** `standards/code/baseline-resolution.md`
- **Rule:** `never-default-to-head` — error
- **Files:**
  - `knowledge/_mcp/lib/git-ops.js:142`
  - `knowledge/_mcp/tools/drift/baseline.js:58`
- **Reason:** Regex matched `baseline = "HEAD"` — see rule description
```

Severity values: `error` (must fix before PR merge), `warn` (recommended fix).

Resolution verbs: `applied` (code fixed), `exempted` (write per-rule exception on the standard), `promoted` (intent to tighten the rule — senior review), `dismissed` (with reason).

## `standards-backlog.md` — aspirational sweep findings

Written by `kb_conform` Phase 1 in aspirational mode (triggered when a standard is tightened). Surfaced advisorily by `kb_get` when editing affected files. Same format as `standards-drift.md`, different file.

```markdown
## wikilink-discipline.section-anchors-for-schema-tables

- **Standard:** `standards/knowledge/wikilink-discipline.md`
- **Rule:** `section-anchors-for-schema-tables` — warn
- **Files:**
  - `specs/features/billing.md` references `[[data/schema/postgres]]` without section anchor
  - `specs/flows/checkout.md` references `[[data/schema/postgres]]` without section anchor
- **Note:** advisory — fix opportunistically when next editing the file
```

Resolution: same verbs as `standards-drift.md`. Promotion is the typical close for backlog entries that have been sitting unresolved.

## `drift-log/YYYY-MM.md` — append-only audit

Written by `kb_drift` and `kb_conform` Phase 2 on every resolution. One markdown file per month. Existing example: [drift-log/2026-04.md](drift-log/2026-04.md).

Event types you'll see in the log:

| Event heading | Source | Meaning |
| ------------- | ------ | ------- |
| `## YYYY-MM-DD · RESOLVED · code→kb` | drift Phase 2 | Code-drift entry closed with summaries / reverted |
| `## YYYY-MM-DD · RESOLVED · kb→code` | drift Phase 2 | KB-drift entry closed with kb_confirmed |
| `## YYYY-MM-DD · CONFORMED · applied` | conform Phase 2 | Standards-drift entry closed by fixing code |
| `## YYYY-MM-DD · CONFORMED · exempted` | conform Phase 2 | Standards-drift entry closed by writing rule exception |
| `## YYYY-MM-DD · DISMISSED · <queue>` | drift / conform | Entry dismissed with reason |
| `## YYYY-MM-DD · DISMISSED-CONFORM` | conform | Standards entry dismissed |
| `## YYYY-MM-DD · ACKNOWLEDGED · <queue>` | both | Entry acknowledged by a reviewer without closing |
| `## YYYY-MM-DD · CLOSED-PROMOTION` | conform | Senior dev closed a `promoted` entry after review |
| `## YYYY-MM-DD · AUTO-CLOSED-PROMOTION · rule changed` | conform | Rule fingerprint changed; entry auto-closed |
| `## YYYY-MM-DD · AUTO-CLOSED-PROMOTION · standard removed` | conform | Standard file removed; entry auto-closed |
| `## YYYY-MM-DD · AUTO-CLOSED-PATTERN-CHANGED` | drift | `_rules.md` pattern fingerprint changed; entry auto-closed |
| `## YYYY-MM-DD · AUTO-DISMISSED · standard removed` | conform | Standard removed; non-promoted entries auto-dismissed |
| `## YYYY-MM-DD · BASELINE-UPDATED` | drift | Baseline ref changed between runs |

Grep recipes:

```bash
# Every code-drift resolution in April
grep -A4 "RESOLVED · code→kb" sync/drift-log/2026-04.md

# Every auto-close event (rule churn audit)
grep "AUTO-CLOSED\|AUTO-DISMISSED" sync/drift-log/*.md

# Every dismissal (intentional drift audit)
grep -A2 "DISMISSED" sync/drift-log/*.md
```

## `review-queue.md` — git merge conflicts on KB files

Written by the `kb-conflict` merge driver. Free-form entries — one section per conflict, with the conflict markers preserved and a one-line summary at the top. Example:

```markdown
## specs/features/billing.md

**Conflict:** semantic merge between `feature/billing-v2` and `main`.
Both branches edited the `## Business rules` section. Manual resolution required.

(conflict markers follow — resolve in your editor)
```

Resolution is a normal git merge resolution; no MCP tool closes these.

---

## `import-review.md` — unclassified import chunks

Written by `kb_import` (auto-classify mode) when a chunk cannot be confidently mapped to a target file. Free-form per-chunk entries with the source document, page/section, suggested targets (ranked), and the chunk text. Resolve by re-running `kb_import` with explicit `target` overrides or by writing the file manually.
